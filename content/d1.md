# D1: A Database at the Edge

The obvious choice for serverless SQL is also the worst one for serverless SQL. Traditional databases are designed to sit in one place, accepting connections from nearby application servers over a persistent network link. Build on the edge — where your compute might run in any of Cloudflare's 300+ locations — and suddenly "connect to your database" means a round trip to a single origin, wiping out any latency advantage you gained by running close to the user. D1 is Cloudflare's answer: a fully managed SQL database designed from the beginning to live alongside Workers, not across the internet from them.

D1 uses SQLite as its query engine. This is not an obvious choice, but it's a deliberate one.

---

## How It Works

**SQLite is everywhere, but usually embedded.** It runs in mobile apps, browsers, and desktop software — wherever you need a self-contained database that doesn't require a server process. D1 takes that engine and makes it a managed cloud service, handling persistence, replication, and backups while preserving the full SQLite SQL dialect you already know. Standard SQL: `SELECT`, `INSERT`, `JOIN`, window functions, full-text search, JSON functions. If it works in SQLite, it works in D1.

**The binding model is what makes it feel different.** A Worker accesses D1 through a binding — a direct reference declared in the project config, not a connection string. When the Worker runs, the database is already there, no TCP handshake, no authentication flow. You call `env.DB.prepare("SELECT * FROM users WHERE id = ?").bind(userId).first()` and you get back a row. The query executes inside Cloudflare's infrastructure, close to where the Worker is running.

**Reads scale geographically.** D1 has a primary database that handles writes and replicates out to **read replicas** in additional Cloudflare locations. Workers are automatically routed to the nearest replica for read queries. This makes read-heavy workloads — fetching content, looking up user profiles, serving product data — fast across the globe. Writes always go to the primary. The consistency model is worth understanding precisely: D1 provides **sequential consistency within a session** — a Worker that writes and then reads will always see its own writes — but cross-session reads from replicas may lag behind the primary by a subsecond window. This is not the same as eventual consistency in the traditional sense: there is a real ordering guarantee, just not strict linearizability across all clients simultaneously.

**The write model is single-threaded per database.** D1 uses SQLite in WAL (write-ahead log) mode: every committed transaction appends frames to the WAL file, which Cloudflare's Storage Relay Service (SRS) immediately forwards to five follower machines across different datacenters. A write is confirmed once at least three followers acknowledge receipt — a quorum-based durability guarantee. Because WAL mode serializes writes through a single writer, there's no concurrent multi-writer model. For most web applications this isn't a problem — even busy apps rarely need thousands of simultaneous writes — but it shapes how you think about schema design under high write load. D1 is not the right tool for a database that receives thousands of writes per second on a single table.

**The multi-tenancy pattern is genuinely unusual.** Most databases assume one big database with many tables. D1 invites you to create thousands of small databases — one per user, one per customer, one per project. An account on the paid plan can have up to 50,000 databases, each up to 10 GB, totaling 1 TB of storage. Pricing is based on rows read and written, not per database, so this costs the same as one database at the same total query volume. Isolating tenants at the database level means no schema collisions, no noisy neighbors, and trivial deletion when a user leaves.

**Time Travel** is D1's built-in disaster recovery feature. Every database maintains a 30-day window of point-in-time recovery — you can restore a database to any minute within that window with a single command. This isn't a separately configured backup job; it's always on.

**Pricing for typical workloads is remarkably low.** The paid plan includes 25 billion rows read and 50 million rows written per month before any charges apply. After that, reads cost $0.001 per million and writes cost $1.00 per million. Storage is $0.75/GB/month beyond the 5 GB included. No egress charges. For a content-heavy site that reads far more than it writes, this often means D1 costs nothing at all in a given month.

## In the Wild

**Per-user data isolation.** A SaaS product stores each customer's data in its own D1 database — 10,000 customers, 10,000 databases. Every query is naturally scoped to one tenant with no `WHERE tenant_id = ?` clause needed, no risk of a buggy query leaking rows across accounts, and instant deletion when a subscription ends. The architecture that would require a complex sharding layer in PostgreSQL is just the default in D1.

**CMS and content delivery at the edge.** A developer blog or documentation site stores its posts in D1. A Worker serves every page request: query D1 for the article, render it, and return it — all within Cloudflare's network, before the request ever touches an origin server. Response times in the tens of milliseconds, globally, with no separate caching layer needed for database results.

**Serverless backends for mobile apps.** A small team ships an iOS app backed entirely by Workers and D1. User accounts, preferences, and activity history live in D1. No backend servers to provision, no RDS instance to right-size, no VPC to configure. The database scales to zero when users are asleep and handles burst traffic automatically.

**Event tracking and lightweight analytics.** A marketing site logs form submissions, click events, and A/B test results to D1 via a Worker. The write volume stays well within the free tier. A separate Worker queries the same database for dashboard views, running aggregation queries against the stored events. No external analytics service required.

## What It Doesn't Do

D1 maxes out at 10 GB per database — this limit cannot be raised. If your application needs a single database larger than that (large media catalogs, extensive audit logs, bulk datasets), you'll need a different tool. The multi-database pattern helps for some use cases, but it's not a substitute for a general-purpose database on large workloads.

D1 is also not built for write-heavy workloads that need high concurrency. A queue that receives thousands of inserts per second, or a time-series store that writes every sensor reading — those workloads want something with a multi-writer model. For transient state and coordination, look at [KV](#/kv) or [Durable Objects](#/durable-objects).
