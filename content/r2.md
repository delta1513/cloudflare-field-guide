# R2: Object Storage Without the Exit Tax

Every file your application saves — images, videos, backups, model weights, log archives — needs somewhere to live. For years, Amazon S3 set the standard: reliable, scalable, available everywhere. But AWS discovered something clever: they could make storage itself cheap while charging handsomely every time data left. *Egress fees.* Move a terabyte out of S3, pay $90. Build a video streaming platform? Those fees stack into something you can't ignore.

Cloudflare R2 is the direct response: object storage with the same interface as S3, the same reliability, and zero egress fees. Not "lower" — zero.

---

## How It Works

**The economics are different because the network is different.** Traditional cloud providers run data centers and buy internet bandwidth like any large business would. Cloudflare's network operates differently — they've spent fifteen years building direct peering relationships with ISPs and carriers in hundreds of cities. Data flowing through Cloudflare's backbone doesn't incur the same transit costs that flow through to AWS's pricing model. That's the structural reason egress fees can be zero.

**The API is deliberately familiar.** R2 implements the S3 API — the same specification AWS created. Any tool, library, or SDK built for S3 works with R2 without code changes. You swap the endpoint URL and credentials; the rest of your code stays the same. `boto3`, the AWS CLI, `@aws-sdk/client-s3` — all work out of the box.

**Storage model: buckets and objects.** A bucket is a namespace and a unit of access control. An object is a blob of bytes stored at a key — a path-like string such as `photos/2024/vacation.jpg`. Objects can be up to 5TB. Large objects use multipart upload, which breaks the file into chunks (each up to 5GB, minimum 5MB except the final part) and uploads them in parallel — R2 implements the same multipart API as S3, so SDKs handle this automatically above their configurable threshold.

**Location hints** let you influence where R2 stores your data. You don't get to pick a specific data center, but you can tell R2 where most reads will originate — "Eastern North America," "Europe," "Asia-Pacific" — and it optimizes placement accordingly. This matters for latency on workloads that aren't fronted by a CDN.

**Workers integration is the interesting part.** Cloudflare Workers — their serverless runtime — can read and write R2 objects with direct bindings. No HTTP call, no authentication, no network hop: the Worker and the storage run in the same environment. The binding exposes an `R2Bucket` object with methods like `get()`, `put()`, `delete()`, and `list()`, plus `createMultipartUpload()` for large objects. `get()` returns an `R2ObjectBody` whose `.body` is a `ReadableStream` — you can pipe it directly into a `Response` for zero-copy streaming. This makes R2 composable in ways that S3 isn't. A Worker can transform an object mid-flight, enforce access control, generate presigned URLs, or stream large files to users — all without a server.

**Access patterns:**
- **Workers binding** — internal, zero-latency, no egress cost
- **S3-compatible API** — authenticated HTTP from anywhere
- **Public buckets** — expose a bucket directly to the internet, serve assets via URL
- **Presigned URLs** — time-limited access tokens for private objects, shareable without credentials

## In the Wild

**Replacing S3 behind a Cloudflare CDN.** If you're already using Cloudflare's CDN in front of S3, you're paying egress from S3 to Cloudflare for every cache miss — then Cloudflare serves users for free. Switching origin storage to R2 eliminates that S3 egress bill entirely. For high-traffic media sites, this can be the single largest line item on the AWS invoice.

**Podcast and audio hosting.** Audio files are large, frequently downloaded, and served globally. A podcast with 100,000 listeners downloading a 60MB episode generates 6TB of egress. On S3, that's around $540 for a single episode release. On R2, it's zero. Independent creators and small podcast platforms have moved wholesale, with the cost dropping to the storage charge alone (roughly $0.015/GB/month).

**Machine learning artifact storage.** Training runs produce enormous checkpoints — model weights, datasets, embeddings, evaluation artifacts. Teams often need to experiment freely: save every checkpoint, roll back, compare runs. On S3, every read for comparison or re-evaluation has a cost. On R2, teams can afford to instrument more, save more, and move faster without the cost anxiety.

**Data lakes on a budget.** Analytics pipelines that carefully rationed S3 reads (because every query has egress cost) can now run freely. Store your event logs and raw data in R2, process them with Workers or external compute, join and aggregate without watching the meter.

**Access-controlled file sharing.** Combine R2 with a Worker that checks authentication headers, and you have a private file server with custom logic. The Worker intercepts every request: verify a JWT, log the access, generate a watermarked version on the fly, add custom cache headers — then stream the R2 object to the user. No infrastructure to manage, and the compute lives alongside the storage.

> R2's pricing is simple: $0.015/GB stored per month. Egress: $0. Class A operations (writes, lists): $4.50 per million. Class B operations (reads): $0.36 per million. There's a generous free tier — 10GB storage, 1M Class A ops, 10M Class B ops per month.

## What It Doesn't Do

R2 is object storage, not a file system. There's no concept of directories (though key prefixes simulate them), no locking, no append operations. It's not a database. For structured data with queries, look at [D1](#/d1) (SQL) or [KV](#/kv) (key-value). For video with transcoding and adaptive bitrate, look at [Stream](#/stream).

R2 also does not support object versioning — there's no equivalent to S3's versioning feature that keeps previous copies of an object on overwrite. If you need rollback capability, you have to implement it yourself (e.g., key naming conventions that encode version or timestamp). Object lifecycle rules can auto-delete objects after a configured number of days, but there's no multi-version retention. This is a meaningful constraint for use cases like audit logs or mutable model checkpoints.

R2 also doesn't include a built-in CDN. Serving objects from R2 directly via the public bucket URL means requests hit the nearest Cloudflare location, but objects aren't cached across the network. Put Cloudflare's CDN in front of R2 (which takes a checkbox, since R2 is already a Cloudflare product) and you get global caching on top of globally-available storage.
